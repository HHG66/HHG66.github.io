前言

前端工程化：

为满足前端技术的发展，提出了将前端代码使用工程的思想搭建的一种思路，这样实现规范、标准，能够提高效率降低成本。这是一种必然的结果，前端本身就是软件工程里面的一环。

前端工程化的项目就具有下面几个特性:

模块化:

组件化：

自动化：

工程化的解决方法：早期的grunt、gulp。目前流行的webpack和parcel。目前webpack较多。

webpack就是前端工程化的具体解决方案。

下面笔记是基于webpack5。

webpack简介

首先要明确的是wepack依赖node环境，当我们安装node的时候要注意，如果配置了环境变量，打包的时候就需要手动修改这个环境变量的值，不然会打包成开发环境的包，这样会出现未知的错误。一定要注意！之所以增加这个环境变量是为了减少误操作导致的错误。10月底写新疆环境跳进了这个坑，打包之前一定注意环境变量。

为什么是webpack

第三个问题，选择什么工具，webpack提供的功能是最全，可以自定义插件的方式实现定制。当然随着发展，目前有了vite 新构建工具，未来vite在vue开发中将会变得更加重要。

污染全局作用域问题

每个插件引用一般会在window全局变量挂载一些变量和方法，这样导致大型项目window变量会很臃肿，不同插件之间可能导致变量名冲突等等问题，如何解决这种问题，和jQuery类似webpack使用了自执行函数（立即调用函数）（IIFE）变量名和方法全部挂载到这个函数下，这样就解决了这个问题。

引入机制

第二个要解决的痛点就是如何拆分代码实现代码模块的使用，这个问题node提出common.js，通过require的方式来加载代码的。

需要注意的是早期游览器是不支持require的，如何解决这样的语法呢，早期使用require.js 的库来解决的，但是这样的话就增加了复杂度，直到模块成为ECMAScript的官方标准才解决这个问题，但同时游览器得支持并不完整。所以实际为了解决各种各样得问题还是需要工具来帮助开发。

自然得引入插件就存在两种处理机制，一种是通过webpack本身内置支持处理方式处理，一种是插件处理的机制。

web works

Web Worker 的作用，就是为 JavaScript 创造多线程环境，允许主线程创建 Worker 线程，将一些任务分配给后者运行。在主线程运行的同时，Worker 线程在后台运行，两者互不干扰。等到 Worker 线程完成计算任务，再把结果返回给主线程。这样的好处是，一些计算密集型或高延迟的任务，被 Worker 线程负担了，主线程（通常负责 UI 交互）就会很流畅，不会被阻塞或拖慢。（来自阮一峰）

Tree Shaking

tree shaking 是一个术语，通常用于描述移除 JavaScript 上下文中的未引用代码(dead-code)。它依赖于 ES2015 模块语法的 静态结构 特性，例如 import 和 export。这个术语和概念实际上是由 ES2015 模块打包工具 rollup 普及起来的。

sideEffects

在trss Shaking中有一些代码是由副作用的，webpack是不能处理这些有副作用的代码，所以在webpack4中默认是全部文件具有副作用，如何描述那些文件和代码具有副作用，通过sideEffects来设置。

sideEffects可以设置true或者false，分别代表全部文件不具有副作用和全部文件具有辅作用，同时还可以接受一个数组来指定文件是否不具有副作用。
